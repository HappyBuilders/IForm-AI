# IForm-AI 排查手册（Troubleshooting Playbook）

> 用途：当用户问“为什么不生效 / 为什么异常 / 帮我排查”时，优先按本手册走固定排查路径，减少自由发挥式检索带来的耗时与漂移。

---

## 总原则

1. **先判断问题类型，再读文档，不要先全库搜索。**
2. **先判断“数据有没有写进去”，再判断“为什么没显示出来”。**
3. **先看真实数据，再看配置解释。**
4. **同类问题优先使用固定排查顺序，避免每次重新设计路径。**

---

## 标准排查输入

如果用户已经提供以下信息，优先利用：

- 环境：test / daily / pre / c1 / c2 / c3 / c4
- `pkBo`
- `pkBoins`
- 现象描述
- 报错信息
- 截图 / 接口返回

如果用户没有提供完整信息，也先按本手册给出第一轮判断，不必立刻索要全部参数。

---

# 一、通用排查总流程

适用于绝大多数“表单不生效 / 回写异常 / 展示异常 / 提交异常”问题。

## Step 1：先归类问题

优先把问题归入以下 6 类之一：

1. **回写类**
   - 关键词：回写、没带出、没同步、控件没值、节点意见不显示
2. **提交类**
   - 关键词：提交失败、无反应、发起审批失败、流程没触发
3. **权限类**
   - 关键词：只读、不可编辑、看不到、节点无法修改
4. **显示类**
   - 关键词：显示异常、列表详情不一致、显示 ID、不显示名称
5. **同步类**
   - 关键词：PC/移动端不一致、复制后异常、刷新后值不对
6. **接口/集成类**
   - 关键词：接口报错、开放接口、系统集成、数据对接

## Step 2：先读快速索引

先读：
- `references/QUICK_INDEX.md`

按关键词命中高频问题，再决定只读哪 1~3 篇文档。

## Step 3：优先判断“真实数据层”还是“页面展示层”

先区分：

### A. 数据层问题
表现：
- 提交后接口返回里也没有值
- 流程记录中也没有对应数据
- 业务字段根本没更新

常见原因：
- 没有真的写入
- 映射错字段
- 节点权限阻止写入
- 流程动作未触发
- 脚本/规则未执行

### B. 展示层问题
表现：
- 接口返回里有值
- 但页面控件没显示 / 显示错 / 显示空

常见原因：
- 控件绑定字段错误
- 显示字段配置错误
- 表单版本不一致
- 读的是旧缓存
- 返回的是 ID，但控件要显示名称

## Step 4：只读对应文档，不扩散

- 证据足够就停
- 只在当前分类证据不足时，才扩展到相邻分类
- 不要在第一轮就把 `forms/`、`troubleshooting/` 全扫一遍

# 二、专项排查模板

## 模板 A：审批意见 / 回写不到控件上

### 适用现象
- 审批意见控件没有内容
- 某些节点意见没显示
- 流程监控临时加签后的意见不显示
- 审批完成后正文控件没回写

### 优先读取
1. `references/QUICK_INDEX.md`
2. `references/forms/01-表单设计与控件配置/审批意见.md`
3. `references/forms/03-审批流程与节点操作/3.2 审批过程中的字段权限问题.md`
4. `references/forms/04-表单数据展示与同步/4.2 数据同步问题.md`
5. `references/forms/04-表单数据展示与同步/4.1 数据显示异常问题.md`

### 固定判断顺序

#### 1）先判断：审批意见写到哪里了？
区分两种：
- 写到了**流程审批记录**
- 写到了**业务表单字段**

很多问题本质是：
> 审批意见只进了流程记录，没有自动映射到表单控件字段。

#### 2）再判断：控件绑定字段是否正确
检查：
- 表单控件字段编码
- 审批意见配置里的环节绑定
- 回写目标字段编码
- 是否主表/子表路径不一致

#### 3）再判断：流程环节是否变更过
重点排查：
- 历史版本配置错误
- 流程环节调整后未重新绑定
- 活动节点重新设计后未维护审批意见绑定
- 流程监控临时增加的环节未勾选对应选项

#### 4）再判断：当前节点字段权限
检查：
- 当前节点该字段是否可写
- 是否被权限规则设为只读/隐藏
- 是否审批动作时字段被过滤

#### 5）最后判断：显示层问题
如果接口里有值，页面没显示，优先看：
- 显示字段配置
- 缓存 / 版本差异
- 列表字段与详情字段不一致

### 默认结论模板
如果证据不足，优先给出：

> 高概率不是控件本身坏了，而是审批意见只写入了流程意见记录，没有同步到业务字段；其次再排查字段绑定、节点权限、流程环节重绑和显示配置问题。

## 模板 B：提交失败 / 流程没触发

### 优先读取
1. `references/QUICK_INDEX.md`
2. `references/forms/02-表单提交与流程触发/2.1 表单提交异常问题.md`
3. `references/forms/02-表单提交与流程触发/2.2 流程触发规则问题.md`
4. `references/forms/02-表单提交与流程触发/2.3 校验规则问题.md`

### 固定判断顺序
1. 是否前端校验已拦截
2. 是否真正触发提交接口
3. 是否流程规则未命中
4. 是否节点条件/校验不通过
5. 是否后端保存成功但流程创建失败

---

## 模板 C：字段不可编辑 / 节点无法操作

### 优先读取
1. `references/QUICK_INDEX.md`
2. `references/forms/03-审批流程与节点操作/3.2 审批过程中的字段权限问题.md`
3. `references/forms/03-审批流程与节点操作/3.1 审批环节无法继续执行问题.md`
4. `references/forms/06-权限控制与数据查看/6.1 权限配置问题.md`

### 固定判断顺序
1. 是表单权限还是流程节点权限
2. 是查看不到还是编辑不了
3. 只某个节点异常，还是全流程都异常
4. 是否分支流程权限配置不同

---

## 模板 D：数据显示异常 / 显示 ID 不显示名称

### 优先读取
1. `references/QUICK_INDEX.md`
2. `references/forms/04-表单数据展示与同步/4.1 数据显示异常问题.md`
3. `references/forms/01-表单设计与控件配置/关联控件关联字段.md`
4. `references/forms/04-表单数据展示与同步/4.2 数据同步问题.md`

### 固定判断顺序
1. 接口里返回的是 ID 还是名称
2. 控件显示字段配置是否正确
3. 列表和详情是否取自同一字段
4. 是否有表单版本变更/旧字段删除

---

## 模板 E：PC / 移动端不一致

### 优先读取
1. `references/QUICK_INDEX.md`
2. `references/forms/04-表单数据展示与同步/4.2 数据同步问题.md`
3. `references/forms/04-表单数据展示与同步/4.1 数据显示异常问题.md`

### 固定判断顺序
1. 是否一端本地缓存未刷新
2. 是否移动端有单独修改逻辑
3. 是否复制单据导致老字段残留
4. 是否版本切换后结构不一致

# 三、回答生成规范

## 当证据充足时
输出应包含：
1. 现象归类
2. 最可能原因
3. 排查依据（来自哪几篇文档 / 哪类逻辑）
4. 建议用户下一步检查什么

## 当证据不足时
不要继续盲目扩展检索，优先输出：
1. 当前高概率判断
2. 还差什么关键证据
3. 用户补什么信息后可以继续收敛

推荐补充项：
- `pkBo`
- `pkBoins`
- 环境
- 审批节点截图
- 接口返回 JSON
- 字段编码

---

# 四、什么时候优先看真实接口数据

遇到以下情况时，优先看 H5 页面/接口数据，而不是继续读文档：

1. 用户已经给了 `pkBo / pkBoins / 环境`
2. 问题是“为什么没回写 / 为什么没显示 / 为什么不一致”
3. 同一个问题可能同时涉及配置、权限、显示、缓存多个层次

原则：
> 先看真实返回，再用文档解释返回结果。

---

# 五、这份手册的目的

- 降低自由检索带来的耗时
- 提高高频问题的首轮命中率
- 让排查回答更稳定、更像“固定诊断流程”而不是临场搜索
- 把“先找资料”改成“先判断问题类型 → 直接读命中文档 → 快速给结论”
