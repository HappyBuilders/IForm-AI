# 智能业务表单系统详情页数据获取 技术方案

## 1. 背景与目标
 当前系统中通过首页的参数输入，还带入到详情页面进行查询数据展示，所以需要根据参数来调用接口获取数据渲染。
**目标**：从业务系统获取各页签需要的数据进行渲染

## 2. 核心设计思路：sso或yht_access_token授权 + 调用业务系统接口获取

1. **统一授权**：目前系统首页有两种授权方式，sso的登录授权（暂时未实现），yht_access_token 的授权直接入参传入。所以后续调用业务系统接口，都需要在接口请求头中加入授权的yht_access_token
2. **接口调用**：每个页签的数据并不是独立进行接口调用渲染的，需要调用几个接口后进行数据汇总提取关键信息进行渲染。

 #### 3.1.1 单据数据信息（参数解析）
  1. 请求参数：接口入参里的pk_bo、pk_boins对应入参的表单id和单据id参数，请求头需要带入yht_access_token。具体展示的调用地址需要工具首页设置的环境+接口地址(配置文件中：单据数据信息)
  2. 返回参数：返回数据中data中是返回数据的json字符串内容。
      head 节点下的数据是主表数据，body下是子表数据（bodys里存了每行子表的数据）。表单时一个动态的配置表单（也就是沒有固定的字段属性，需要进行解析表单配置时设计的字段信息，才能渲染单据数据）
      其中每一个key类似“20260422140244h6vpm6cX41” 是一个控件的fieldId 需要根据 "表单配置信息" 的返回数据才能进行解析渲染。
  3. 返回的是表单字段的权限processauthinfo（fillIn是填单页的权限，类似“approveUserTask_33a4ddbd73144877a47076339a3094f9” 的是对应审批环节的权限）auth 0是隐藏，1是查看，2是编辑

#### 3.4.1 功能说明
     1. 这个页签的功能和其它页签的功能有点区别，它的接口数据不是从对应环境的业务系统取数据，而是从一个没有环境区别的jira系统来获取数据。但也需要走代理。
     2，这个页签的接口认证和其他页签不一样，不是走统一的yht_access_token认证，而是需要自己系统的cookie认证。
#### 3.4.2 功能改造
     1.这个页签页面功能需要简单的改造，首先这个页签支持在该页签中界面设置两个参数进行后续该页签的接口调用使用：“Jira工单号”和“Jira系统Cookie”.
     2，”Jira工单号“这个参数需要从首页就支持设置，“Jira系统Cookie”这个参数不用在首页维护，仅在页签中展示录入就行，是一个必填参数，并和当前页签做联动录入和修改维护。区别在于首页上是非必填。进入当前页签“Jira问题分析”页签后是必填，因为这个页签后续接口使用。
     3.该页签主要做几个功能，第一展示当前入参工单号的基本信息和详情。第二再出一个与当前工单有关联的工单列表（这个功能需要将获取jira数据渲染），第三预留一个智能分析的区域，后续集成其他的智能体或skill来进行智能分析。
     4，修改这个页签的接口调用逻辑，不在使用其他页签的请求头携带yht_access_token的方式而是请求头中携带自己页签的cookie.
     5.改造接口的url。不在是环境+接口地址。而是域名指定到配置文件中不随环境切换（没有环境区分就一个）

#### 3.4.4 当前实现口径（2026-04-26）
##### 3.4.4.1 参数与存储策略
1. 首页已支持可选参数 `Jira工单号`，用于带入详情页 Jira 页签。
2. `Jira系统Cookie` 不在首页录入，仅在详情页 `Jira问题分析` 页签中维护。
3. `Jira系统Cookie` 不写入 URL，不进入详情页顶部参数摘要，只保存到浏览器 `sessionStorage`。
4. 页签内提供“加载 Jira 数据”和“清空 Cookie”按钮，支持当前页签内联动录入与刷新。
5. 如果用户粘贴的是 `Cookie: xxx` 形式的完整文本，前端和代理都会自动归一化为真实 Cookie 值。

##### 3.4.4.2 请求链路与代理实现
1. Jira 页签不直接从浏览器访问 `https://gfjira.yyrd.com`，而是统一走本地代理中转。
2. 当前已落地的代理接口为：
   - `POST /api/jira/issue-table`
   - `GET /api/jira/issue-detail`
3. 前端通过请求头 `x-jira-cookie` 传递 Jira Cookie，代理再转换为真实 `Cookie` 请求头转发给 Jira 系统。
4. 代理内部固定 Jira 基础地址，不跟随首页环境切换：
   - 列表接口：`/rest/issueNav/1/issueTable`
   - 详情接口：`/secure/AjaxIssueEditAction!default.jspa`
5. 代理额外补充 Jira 请求头：
   - `x-atlassian-token: no-check`
   - `x-requested-with: XMLHttpRequest`
   - `__amdmodulename: jira/issue/utils/xsrf-token-header`

##### 3.4.4.3 查询与加载策略
1. Jira 列表查询条件使用配置文件中的 JQL 模板：
   `(project = UPESN AND issuetype = 支持问题 AND 领域模块 in cascadeOption(10707, 10710) AND status = 支持确认完成) OR issueKey = {issueKey} order by created DESC`
2. 发起查询前，将 `{issueKey}` 替换为当前页签录入的 `Jira工单号`。
3. 列表请求的附加参数当前包括：
   - `startIndex=0`
   - `layoutKey=split-view`
4. 返回结果中使用 `issueTable.table` 渲染“近期工单列表”。
5. 前端会在 `issueTable.table` 中优先匹配 `key === Jira工单号` 的记录，识别当前工单，并取其 `id` 继续查询 Jira 详情接口。
6. 详情页进入后，核心页签先加载：
   - 表单配置信息
   - 单据数据信息
   - 流程审批信息
   - 业务日志
7. `Jira问题分析` 页签改为异步后置加载，不再阻塞其它页签首屏渲染。
8. Jira 缺少 Cookie、缺少工单号或 Cookie 失效时，只影响 Jira 页签自身，不影响其它页签渲染。

##### 3.4.4.4 页面结构与展示顺序
1. 当前 Jira 页签按以下顺序展示：
   - 当前工单基础信息
   - 工单详细内容
   - 相似场景工单列表
   - 智能分析预留
   - 近期工单列表
   - 原始返回数据
2. 原先的“当前工单概述”区域已取消，避免与“当前工单基础信息”重复。
3. 原先的“关联工单列表”已更名为“近期工单列表”，明确该区域只是当前查询条件下的近期集合，并不表示真实关联。
4. `相似场景工单列表` 为独立预留区域，后续用于接入其他 skill、智能体或大模型分析能力，不复用近期工单列表数据。
5. “近期工单列表”已移动到“智能分析预留”区域下方展示。

##### 3.4.4.5 字段解析与渲染规则
1. Jira 详情接口返回的 `fields[]` 中，`editHtml` 不能直接原样注入页面，而是通过 DOM 解析有效值后再转成系统内表格渲染。
2. “当前工单基础信息”中不再展示“查询结果总数”。
3. 以下字段已提升到“当前工单基础信息”中重点展示：
   - `剩余处理时长`
   - `SOP单据号`
   - `外部系统编号`
4. “当前工单详细字段”已更名为“工单详细内容”。
5. `到期日` 展示格式统一为：
   - 有时分时：`YYYY-MM-DD HH:mm`
   - 仅日期时：`YYYY-MM-DD`
6. “解决方案”字段必须按精准规则提取，只认 `label === 解决方案` 的字段值，不再从 `AI处理结果`、`处理方案`、`建议措施` 等其它字段兜底推断。
7. Jira 详情字段当前已按白名单控制解析范围，白名单在配置文件中维护，便于后续继续扩展。

##### 3.4.4.6 近期工单详情联动
1. “近期工单列表”中的每一条记录都提供“查看详情”按钮。
2. 点击“查看详情”后，会调用同一个 Jira 详情接口加载该条工单的“工单详细内容”。
3. 近期工单详情区域同样重点展示 `解决方案` 字段。
4. 详情渲染完成后，页面会自动滚动定位到“近期工单详情”区域，避免用户手动长距离滚动查找。

##### 3.4.4.7 错误处理
1. 如果 Jira 返回的是登录页 HTML，而不是业务 JSON，前端统一提示：
   `Jira 请求返回登录页，当前 Jira 系统 Cookie 可能已失效，请重新填写后重试`
2. 代理对 `401/403` 场景会明确返回 Cookie 无效或无权限提示。
3. 登录页识别已收敛为“HTML 特征 + 登录标识”联合判断，避免正常 Jira JSON 中包含 `atlassian-*` 内容时被误判成登录页。
4. 缺少 `Jira系统Cookie` 时，Jira 页签进入待填写状态，而不是抛出阻塞全局加载的错误。
